# 右侧趋势突破策略分组与板块覆盖校验修复需求

## 背景

执行内置策略 `00000000-0000-0000-0000-000000000006`（右侧趋势突破综合策略）时，最新一次手动选股任务 `92dbe04e-6a1a-4e17-9f2b-89866ba346a8` 加载有效股票 5335 只，最终入选 0 只。

逐因子统计显示数据并非整体缺失：

- `ma_trend` 通过 265 只
- `breakout` 通过 1333 只
- `sector_rank` 通过 2943 只
- `sector_trend` 通过 5330 只
- `macd` 通过 273 只
- `turnover` 通过 2112 只
- `money_flow` 通过 1334 只，缺失 1 只
- `rsi` 通过 2117 只

按现有 `AND` 顺序叠加后，`ma_trend + breakout + sector_rank + sector_trend` 仍剩 115 只，但继续要求 `macd == true` 后剩 0 只。因此主因是策略逻辑过严，不是关键行情和资金流数据整体缺失。

同时，板块行情覆盖存在独立数据一致性问题：`tushare_import_log` 显示 `dc_daily / ths_daily / tdx_daily / sw_daily / ci_daily` 在 2026-04-28 至 2026-04-30 多次完成且有 `record_count`，但 TimescaleDB `sector_kline` 实表各来源最新日期均停留在 2026-04-24。每日快速链路当前基于导入日志跳过板块日行情，导致 2026-04-29 和 2026-04-30 的板块行情缺口没有被识别和补导。

2026-05-02 已优先单独处理板块行情缺失问题，实际发现并修复了以下落库链路问题：

- 板块日行情接口经字段映射后使用 `sector_code`，但 `sector_kline` 写入逻辑只读取 `ts_code`，导致映射后的有效行被跳过。
- `sector_kline.time` 当前数据库列为 `timestamp without time zone`，而 Tushare 时序时间规范化后为 UTC aware `datetime`，触发 asyncpg `offset-naive and offset-aware` 写入错误。
- 每日快速/缺口补导收尾阶段使用全量 `kline` 大表完整性扫描，容易卡住工作流完成状态。
- `gap_repair` 在计划缺口为空时错误回退到 daily-fast 全链路，可能误触发大规模导入。

修复后已补导 `20260425` 至 `20260430` 板块日行情，`sector_kline` 实表目标日 `2026-04-30` 覆盖为：`DC=1013`、`THS=1504`、`TDX=615`、`TI=439`、`CI=437`，零缺口验证工作流完成且 `missing_key_groups=[]`。

## 术语

- 主条件：由策略配置显式标识的硬性筛选条件，用于保证选股结果符合策略定位。不同策略模板可以自由选择哪些因子属于主条件。
- 确认因子：由策略配置显式标识的增强信号，可以按 OR、至少 N 个通过、仅加分等模式参与筛选。不同策略模板可以自由选择哪些因子属于确认因子。
- 因子角色：策略配置中对单个因子的用途标识，例如 `primary`、`confirmation`、`score_only`、`disabled`，由策略模板或交易员配置决定，不由引擎硬编码。
- 因子筛选统计：一次智能选股任务完成后，每个条件因子的独立评估结果统计，包括通过数、缺失数、参与评估数，并可扩展累计筛选后剩余数。
- 实表覆盖：以最终业务读取表中的目标交易日行数和最新日期为准，例如 `sector_kline`。
- 导入日志覆盖：以 `tushare_import_log` 中成功记录推断接口已执行，不代表最终业务表一定可用。

## 需求

### 1. 策略应支持可配置的主条件 + 确认因子分组

**用户故事：** 作为量化交易员，我希望每个策略模板都可以自由标识哪些因子是主条件、哪些因子是确认因子、哪些因子只参与加分，避免系统把固定因子硬编码为某种角色，同时避免所有信号同一天强制共振导致无股票入选。

#### 验收标准

1. WHEN 策略配置包含因子角色 THEN 系统 SHALL 按配置中的 `primary`、`confirmation`、`score_only` 等角色执行，不得在策略引擎中硬编码固定因子属于主条件或确认因子。
2. WHEN 交易员编辑策略模板 THEN 系统 SHALL 允许对任意已启用条件因子标识其角色，例如将 `ma_trend` 设为主条件、将 `macd` 设为确认因子，或反过来按策略需要调整。
3. WHEN 策略配置包含主条件组 THEN 系统 SHALL 支持主条件组使用 `AND`、`OR`、至少 N 个通过等组合逻辑，默认建议为 `AND`。
4. WHEN 策略配置包含确认因子组 THEN 系统 SHALL 支持确认因子组使用 `OR`、至少 N 个通过、仅加分不拦截等组合逻辑。
5. WHEN 执行右侧趋势突破综合策略 THEN 系统 SHALL 不再要求 `breakout == true` 与 `macd == true` 必须作为同级硬性 `AND` 条件，除非该策略模板显式这样配置。
6. WHEN 内置右侧趋势突破综合策略首次初始化 THEN 系统 SHOULD 提供推荐配置：趋势、突破、板块强度作为主条件，MACD、RSI、资金流作为确认因子；但该推荐配置 SHALL 可被交易员在策略模板中修改。
7. WHEN 股票满足主条件且确认因子按配置通过 THEN 系统 SHALL 将股票纳入候选结果；WHEN 确认因子配置为仅加分 THEN 系统 SHALL 不因确认因子未通过而直接剔除股票。
8. WHEN 生成结果评分 THEN 系统 SHALL 保留因子角色、分组通过情况和评分来源，便于复盘。
9. WHEN 旧策略配置仍使用平铺 `factors + logic` 结构 THEN 系统 SHALL 保持向后兼容，不破坏其他策略模板。

### 2. 策略配置应表达因子角色、分组和确认模式

**用户故事：** 作为策略配置维护者，我希望策略配置可以表达“哪些因子属于哪个角色、每组如何组合、确认因子是否拦截”的结构，而不是只能用全局 AND/OR。

#### 验收标准

1. WHEN 策略配置包含因子分组结构 THEN `StrategyConfig.from_dict()` 和 `to_dict()` SHALL 能正确反序列化和序列化。
2. WHEN 策略配置不包含因子分组结构 THEN 策略引擎 SHALL 继续按现有 `logic` 字段执行。
3. WHEN 前端或 API 获取策略模板 THEN 响应 SHALL 包含新的分组配置，不丢失旧字段。
4. WHEN 用户仍使用因子编辑器的普通平铺因子 THEN 系统 SHALL 不要求用户立即迁移。
5. WHEN 因子编辑器展示策略条件 THEN 系统 SHALL 允许交易员为每个条件因子选择角色和所属分组。
6. WHEN 一个因子未显式配置角色 THEN 系统 SHALL 使用兼容规则从旧 `logic` 推导角色，且在配置保存后不改变原有筛选语义。

### 3. 内置右侧趋势突破综合策略应更新默认配置

**用户故事：** 作为量化交易员，我希望内置策略模板默认采用更符合交易逻辑的配置，执行后能产生合理候选，而不是因为过度共振条件长期为空。

#### 验收标准

1. WHEN 系统初始化或读取内置策略 `00000000-0000-0000-0000-000000000006` THEN 其默认配置 SHALL 使用可编辑的主条件和确认因子组。
2. WHEN 数据库中已存在旧版内置策略配置 THEN 系统 SHALL 提供兼容迁移或启动时修正机制，避免旧配置继续硬 AND。
3. WHEN 执行该策略 THEN 日志 SHALL 输出主条件通过数、确认因子通过数、最终入选数，便于后续复盘。
4. WHEN 交易员调整该策略模板的因子角色 THEN 系统 SHALL 按最新模板配置执行，不应被内置默认值覆盖。

### 4. 每日快速计划必须校验 `sector_kline` 实表覆盖

**用户故事：** 作为量化交易员，我希望一键导入的完整性检查以最终使用的数据表为准，避免导入日志显示完成但实际板块行情仍缺失。

#### 验收标准

1. WHEN 每日快速链路检查 `dc_daily` THEN 系统 SHALL 查询 TimescaleDB `sector_kline` 中 `data_source='DC' AND freq='1d'` 的目标交易日覆盖。
2. WHEN 检查 `ths_daily`、`tdx_daily`、`sw_daily`、`ci_daily` THEN 系统 SHALL 分别按 `THS`、`TDX`、`TI`、`CI` 查询 `sector_kline` 实表覆盖。
3. WHEN 导入日志显示目标交易日成功但 `sector_kline` 目标交易日覆盖为 0 THEN 系统 SHALL 判定该接口未覆盖，不得跳过导入。
4. WHEN `sector_kline` 最新日期早于目标交易日 THEN 完整性摘要 SHALL 展示实表最新日期、目标日期覆盖数和修复建议。
5. WHEN 实表覆盖不足 THEN “补齐缺口”按钮 SHALL 可用，并应包含对应板块日行情接口。
6. WHEN 每日快速或缺口补导完成 THEN 系统 SHALL 使用轻量目标日覆盖检查完成工作流收尾，不应再对 `kline` 等大表执行无边界全量扫描。
7. WHEN 缺口补导计划中 `missing_steps=[]` THEN 系统 SHALL 完成轻量完整性检查并直接结束，不得回退执行 daily-fast 全链路。

### 5. 导入任务完成后应进行目标日实表反查

**用户故事：** 作为数据维护者，我希望板块日行情导入任务完成后自动反查实际写入情况，避免 API 返回行数被误认为业务表可用。

#### 验收标准

1. WHEN `dc_daily / ths_daily / tdx_daily / sw_daily / ci_daily` 导入完成 THEN 系统 SHALL 反查 `sector_kline` 中请求日期范围内的实际覆盖日期和行数。
2. WHEN API 返回 `record_count > 0` 但目标日期实表覆盖为 0 THEN 导入结果 SHALL 标记为失败或数据不完整，并写入可读错误信息。
3. WHEN 目标日期不是交易日 THEN 系统 SHALL 使用交易日历回退后的目标交易日进行覆盖判断。
4. WHEN 反查发现只有部分日期写入 THEN 系统 SHALL 在 `extra_info` 中记录缺失日期列表，供缺口补导使用。
5. WHEN 板块行情行经字段映射后包含 `sector_code` 而不包含 `ts_code` THEN `sector_kline` 写入 SHALL 使用 `sector_code` 作为板块代码。
6. WHEN `sector_kline.time` 数据库列为无时区 timestamp THEN 系统 SHALL 将规范化后的 UTC 时间转换为无时区 UTC `datetime` 后写入，避免时区类型不匹配导致落库失败。

### 6. 数据缺失原因应可诊断并给出处理方案

**用户故事：** 作为量化交易员，我希望系统能说明板块行情缺失是 API 无数据、写入失败、字段映射问题、日期解析问题，还是计划阶段误判，而不是只显示“已覆盖”。

#### 验收标准

1. WHEN 板块日行情实表缺失 THEN 系统 SHALL 在完整性摘要中区分至少以下原因：导入日志误判、API 返回空、字段映射后无有效时间/代码、TS 写入失败、目标日非交易日。
2. WHEN 当前已有日志显示 `record_count > 0` 但实表没有目标日数据 THEN 系统 SHALL 优先归因为“导入日志与实表覆盖不一致”，并建议重新执行对应接口的缺口补导。
3. WHEN 缺口补导再次完成后仍无实表覆盖 THEN 系统 SHALL 提示检查接口返回行的 `trade_date/ts_code` 字段与 `_normalize_tushare_timeseries_time()` 解析结果。
4. WHEN 系统发现 `sector_kline` 所有来源最新日期一致停在旧日期 THEN 系统 SHALL 在维护建议中提示优先补导默认 DC 板块日行情，再按启用选项补导 THS/TDX/TI/CI。
5. WHEN 系统发现接口日志成功但实表未更新 THEN 系统 SHALL 在诊断信息中包含字段映射后有效代码字段、时间字段解析结果、数据库时间列类型和最近一次 TS 写入错误摘要。

### 7. 智能选股完成后应展示每个条件因子的筛选统计

**用户故事：** 作为量化交易员，我希望在智能选股执行完成后，能在“一键执行选股”按钮左侧看到当前策略每个条件因子独立筛选出的股票数量和缺失数量，用于判断到底是哪一个因子过严或数据缺失。

#### 验收标准

1. WHEN 智能选股任务完成 THEN 系统 SHALL 为当前策略每个已启用条件因子生成独立筛选统计，至少包含 `factor_name`、展示名称、通过数、缺失数、参与评估数。
2. WHEN 某因子存在数据缺失 THEN 系统 SHALL 在统计中显示缺失数量，例如 `money_flow 通过 1334 只，缺失 1 只`。
3. WHEN 前端展示当前策略区域 THEN 系统 SHALL 在“一键执行选股”按钮左侧显示最近一次执行的因子筛选统计。
4. WHEN 因子较多导致空间不足 THEN 前端 SHALL 使用紧凑列表、横向滚动或折叠展开方式展示，避免遮挡按钮和当前策略名称。
5. WHEN 用户切换策略模板或修改策略条件 THEN 系统 SHALL 清空或标记旧统计为非当前配置，避免误导交易员。
6. WHEN 策略使用主条件/确认因子分组 THEN 统计 SHALL 保留每个因子的角色信息，并可按主条件、确认因子、仅加分分组展示。
7. WHEN 后端返回选股任务结果 THEN API SHALL 包含该任务的因子筛选统计，前端不得只依赖控制台日志或临时调试输出。
8. WHEN 没有执行过选股或任务失败 THEN 前端 SHALL 显示空状态或失败原因，不应展示过期统计。

### 8. 验证和回归

**用户故事：** 作为开发者，我希望本次修复能用自动化测试覆盖策略分组、旧配置兼容、板块覆盖反查和一键导入计划压缩逻辑。

#### 验收标准

1. WHEN 运行策略引擎单元测试 THEN SHALL 覆盖主条件 AND + 确认因子 OR 的通过和失败场景。
2. WHEN 运行旧策略配置测试 THEN SHALL 验证平铺 `factors + logic` 策略行为不变。
3. WHEN 运行工作流服务测试 THEN SHALL 验证 `dc_daily` 等板块日行情以 `sector_kline` 实表覆盖为准。
4. WHEN 导入日志成功但实表缺失 THEN 测试 SHALL 验证计划不会跳过对应导入步骤。
5. WHEN 前端展示完整性摘要 THEN SHALL 显示实表覆盖状态、缺口建议和补齐缺口动作。
6. WHEN 运行 Tushare 导入任务测试 THEN SHALL 验证 `sector_kline` 接受映射后的 `sector_code`，并按当前表结构写入无时区 UTC 时间。
7. WHEN 运行工作流服务测试 THEN SHALL 验证 daily-fast/gap-repair 使用轻量收尾检查，且空缺口补导不会回退执行 daily-fast 全链路。
8. WHEN 运行智能选股前后端测试 THEN SHALL 验证任务结果包含每因子筛选统计，且前端在“一键执行选股”按钮左侧正确展示通过数和缺失数。
